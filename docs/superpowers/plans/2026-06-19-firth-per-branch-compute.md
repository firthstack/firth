# Firth Per-Branch Isolated Compute Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give every project branch its own isolated Fly compute app (storage stays shared, DB already per-branch via Neon), provisioned eagerly at branch-create.

**Architecture:** Make the `resources` table branch-aware (`fly` rows per-branch, `neon`/`s3` project-scoped); provision a Fly app per branch in a 2-resource branch-create saga; deploy + teardown become branch-aware. Reverses the v1 "compute not branched; redeploy" model.

**Tech Stack:** TypeScript/Node, Fastify, vitest (control-plane); the firth CLI; Vite/React (dashboard); InsForge Postgres migrations.

## Global Constraints
- ES modules; relative imports in `control-plane/src`, `cli/src`, tests use `.js` extensions.
- `resources` becomes branch-aware: `fly` rows carry `branch_id` (per-branch, incl. main); `neon`/`s3` rows are project-scoped (`branch_id NULL`). Unique key: `(project_id, branch_id, kind)`.
- A branch = its Neon DB branch + its Fly app + the SHARED bucket. Compute is provisioned EAGERLY at branch-create.
- branch-create is a compensating saga: a failed Fly provision rolls back the Neon branch; never mask the original error.
- Backend errors are static/controlled strings; secrets never logged; `provider_ref` exposed only via the whitelist (now also passing the non-secret `branch_id`).
- All tests run fully local/offline against in-memory fakes (the `fakeDb`/fake-adapter pattern in the reference). The migration backfill is verified live separately.
- Existing projects must keep working: the migration backfills existing `fly` rows to their project's default-branch id.

function fakeDb(seed: Record<string, any[]> = {}) {
  const tables: Record<string, any[]> = { projects: [], branches: [], resources: [], secrets: [], ...seed }
  return { tables, from(t: string) {
    const filters: Array<(r: any) => boolean> = []
    let mode: 'insert' | 'select' | 'update' = 'select'; let payload: any
    const api: any = {
      insert(v: any) { mode = 'insert'; payload = { id: `${t}-${tables[t].length}`, ...v }; tables[t].push(payload); return api },
      update(v: any) { mode = 'update'; payload = v; return api },
      select() { return api },
      eq(c: string, val: any) { filters.push(val === null ? () => false : (r: any) => r[c] === val); return api },
      is(c: string, val: any) { filters.push(val === null ? (r: any) => r[c] == null : (r: any) => r[c] === val); return api },
      async then(res: any) {
        if (mode === 'update') { for (const r of tables[t]) if (filters.every((f) => f(r))) Object.assign(r, payload); return res({ data: [], error: null }) }
        if (mode === 'insert') return res({ data: [payload], error: null })
        return res({ data: tables[t].filter((r) => filters.every((f) => f(r))), error: null })
      },
    }
    return api
  } }
}

ALTER TABLE public.resources

-- Swap the project-scoped unique key for a branch-aware one.
ALTER TABLE public.resources DROP CONSTRAINT resources_project_id_kind_key;

  ON public.resources (project_id, branch_id, kind);

CREATE INDEX idx_resources_branch ON public.resources(branch_id);

-- Backfill: existing fly rows belong to their project's default (live) branch.
UPDATE public.resources r
SET branch_id = (SELECT b.id FROM public.branches b
                 WHERE b.project_id = r.project_id AND b.is_default AND b.archived_at IS NULL)
WHERE r.kind = 'fly' AND r.branch_id IS NULL;

- **Existing projects are backfilled** so live projects (e.g. `first`) keep working without re-provisioning: the migration sets each existing `fly` row's `branch_id` to its project's default (live) branch, so the now branch-scoped `DeployService` resolves the right app on the first post-migration deploy.
